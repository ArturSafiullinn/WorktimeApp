# Security audit

Дата: 2026-07-22

## Уже закрыто в этом проходе

- API больше не открыт без входа: все `/api/*`, кроме `/api/health` и `/api/login`, требуют Bearer-токен.
- Логин перенесен на сервер. Фронт больше не сравнивает пароль с локальным объектом `accounts`.
- Пароли хешируются через `scrypt`. Старые открытые пароли мигрируют в hash при успешном входе пользователя.
- `/api/accounts` больше не возвращает пароли.
- Управление пользователями закрыто ролью `admin`.
- Подразделения, импорт СКУД и изменение постоянных графиков закрыты ролью `admin`.
- Правки табеля разрешены только `admin` и `boss`; `observer` не может писать.
- Начальник на сервере ограничен своей областью: `employee_ids` и `department_ids` проверяются при чтении сотрудников, табеля и правок.
- Создание, удаление и массовое удаление правок табеля проверяют принадлежность `employee_id` текущему начальнику.
- Логин получил базовый rate limit.
- Добавлен сброс пароля по email: токен отправляется письмом, в БД хранится только SHA-256 hash токена, срок действия 60 минут, одноразовое использование.
- Email пользователя хранится в `app_accounts.email`; администратор может заполнить его в карточке пользователя.
- CORS в production должен задаваться через `CORS_ORIGINS`; открытый CORS оставлен только для local/dev.
- Ошибки части критичных API заменены на безопасные пользовательские сообщения без деталей БД.
- JSON body limit вынесен в `JSON_BODY_LIMIT`, по умолчанию 5 MB.
- Импорт СКУД ограничен по количеству строк.
- Docker production image запускает приложение от пользователя `node`, не от root.
- Production compose требует `APP_SECRET` и `CORS_ORIGINS`.

## Остаточные high/medium риски

### High: пакет `xlsx`

`npm audit --omit=dev` показывает high advisories для `xlsx`:

- Prototype Pollution in SheetJS.
- ReDoS in SheetJS.

У npm нет автоматического безопасного фикса для текущего пакета. Рекомендуемая следующая работа: заменить `xlsx` на поддерживаемый парсер или вынести обработку Excel в изолированный серверный импорт с жесткой валидацией формата, размера и колонок.

### High: существующие прод-пароли

Старые пароли будут хешироваться при следующем успешном входе. Но дефолтные и слабые пароли все равно нужно заменить вручную для всех пользователей, особенно `admin`, `observer` и начальников.

### Medium: токены в localStorage

Сейчас токен хранится в `localStorage`. Это лучше старого клиентского логина, но при XSS токен можно украсть. Более безопасный вариант: httpOnly secure cookie + CSRF-защита.

### Medium: аудит действий неполный

Аудит есть для правок табеля. Нужно добавить аудит для:

- изменения пользователей;
- изменения подразделений;
- изменения постоянных графиков;
- импорта СКУД.

### Medium: инфраструктура вне репозитория

Нужно проверить на production-сервере:

- HTTPS включен на внешнем домене;
- PostgreSQL не доступен извне;
- SSH только по ключу, парольный вход выключен;
- настроены и проверены бэкапы БД;
- есть мониторинг 5xx, диска, CPU/RAM и доступности БД.

## Production checklist перед выкладкой

1. Сгенерировать длинный `APP_SECRET` и добавить в prod `.env`.
2. Указать точный production origin в `CORS_ORIGINS`, например `https://worktime.example.ru`.
3. Сменить все дефолтные пароли.
4. Перелогиниться каждым важным пользователем или принудительно сбросить пароли, чтобы plaintext заменился hash.
5. Проверить, что `/api/employees` без токена возвращает `401`.
6. Проверить, что начальник видит только своих сотрудников.
7. Проверить, что `observer` не может создать правку табеля.
8. Запланировать замену `xlsx`.
